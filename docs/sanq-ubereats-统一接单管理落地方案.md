# sanq.ca 与 Uber Eats 统一接单与管理落地方案（含合规边界）

## 1. 目标与范围

### 1.1 目标
在 **sanq.ca 后台**统一进行：
- Website 订单与 Uber Eats 订单的统一接单与管理。
- 门店营业状态、商品上下架状态的实时同步。
- 菜单维护采用「**Uber 价格权威**」策略：
  - 在 SANQ 后台维护一套 Uber 专用价目表；
  - 菜单上传/单品状态更新时，统一使用这套价格；
  - 后续价格调整也在 SANQ 后台修改并推送。

### 1.2 本期范围（MVP）
- ✅ 统一订单池（Website + Uber Eats）
- ✅ 订单状态回传（接单、制作中、完成、取消等）
- ✅ 门店状态同步（营业/暂停接单）
- ✅ 商品状态同步（可售/停售）
- ✅ Uber 专用价格表维护与菜单推送
- ❌ 拒单流程自动化（先不做，保留在 Uber Tablet 手工处理）

## 2. 按 Uber Eats 规则的合规边界（实现约束）

> 说明：具体字段、状态码与回调安全要求以 Uber Developer Portal 文档与商户协议为准，以下为工程落地需要遵守的通用边界。

1. **订单响应时效要求**：
   - Uber 订单需在平台规定时限内确认状态更新；
   - SANQ 需提供告警与超时兜底（例如超时未操作时提醒门店转到 Uber Tablet）。

2. **状态一致性要求**：
   - 一切已回传 Uber 的订单状态应具备审计日志，避免出现门店端与 Uber 端状态不一致。

3. **菜单与价格一致性要求**：
   - 上线到 Uber 的价格、可售状态必须以当前已发布版本为准；
   - 变更后需可追踪“谁在何时改了什么”。

4. **账号与门店隔离**：
   - 多门店时，凭据、菜单、状态与订单必须按 store_id 隔离，避免串店。

5. **失败重试与幂等**：
   - 对 Uber API 的写操作必须设计幂等键与重试策略，防止重复推单或重复状态更新。

## 4. 核心能力设计

### 4.1 统一订单接单工作台

#### 数据模型（建议）
- `order_channel`: `website | uber_eats`
- `channel_order_id`: 渠道订单号
- `store_id`
- `status_internal`: SANQ 统一状态
- `status_channel`: 渠道状态
- `ack_deadline_at`: 渠道要求响应截止时间
- `raw_payload`: 渠道原始报文（脱敏后存储）
- `sync_state`: `pending | synced | failed`
- `sync_error`

#### 状态映射（示例）
- Uber `new` → SANQ `待接单`
- SANQ `已接单` → Uber `accepted`
- SANQ `制作中` → Uber `in_progress`
- SANQ `完成` → Uber `completed`
- 取消按双方状态机映射，冲突时以“最后可接受状态+人工介入”处理

#### 操作规则
- 本期不做 Uber 拒单自动流：
  - 工作台展示“请在 Uber Tablet 操作拒单”提示；
  - SANQ 只做状态读取与展示同步。

### 4.2 门店状态实时同步

- SANQ 改门店状态（营业/打烊/临时暂停）→ 触发 Uber API 更新。
- Uber 侧回调或拉取发现状态差异 → 自动修正并告警。
- 同步策略：
  - **写后核对**：每次推送后立即读回校验；
  - **定时对账**：每 5~10 分钟巡检门店状态一致性。

### 4.3 商品状态实时同步

- SANQ 商品“可售/停售”变更触发 Uber 菜单项 availability 更新。
- 高峰期建议采用增量更新（仅变更项），减少 API 压力。
- 设置节流与批处理窗口（如 10~30 秒）避免频繁抖动。

### 4.4 菜单维护策略（Uber 价格权威）

#### 规则定义
- 在 SANQ 后台新增（或扩展）`price_book_uber`（Uber 专用价目表）。
- 上传 Uber 菜单、单品状态更新时，统一从 `price_book_uber`取价。
- Website 价格与 Uber 价格允许分离，避免互相覆盖。

#### 发布流程
1. 运营在 SANQ 后台维护 Uber 价目表。
2. 点击“发布到 Uber”（支持按门店/时段）。
3. 系统执行：数据校验 → 生成发布版本 → 推送 Uber → 记录结果。
4. 发布成功后，标记 `menu_version_published` 并可回滚到上一版本。

#### 必备保护
- 发布前校验：缺图、缺税码（如有要求）、分类为空、价格异常。
- 灰度发布：先试点门店，稳定后全量。
- 回滚机制：保留最近 N 个成功版本。

## 6. 异常与风控设计

1. **API 失败重试**：指数退避（如 1s/5s/30s），超过阈值转人工。
2. **幂等保障**：以 `channel + channel_order_id + action` 作为幂等键。
3. **消息去重**：Webhook event_id 去重表，防止重复消费。
4. **数据审计**：订单状态、菜单发布、价格修改全链路审计。
5. **降级策略**：当 Uber 接口异常时，提示门店切换 Uber Tablet 手工兜底。

## 7. 运营流程建议

### 日常调价流程
1. 运营在 SANQ 后台修改 Uber 价目表。
2. 选择门店并发布。
3. 系统完成推送并回显发布结果。
4. 失败项自动进入重试队列并通知负责人。

### 高峰保供流程
1. 商品短缺 → 在 SANQ 标记停售。
2. 系统 30 秒内推送 Uber 商品停售状态。
3. 恢复供应后再上架并推送。
